智能招聘评测系统
轻量技术架构设计文档

面向小型内部项目，采用单一 Python 技术栈，删除不必要的微服务、消息队列、缓存、对象存储和独立前端工程，以最低部署复杂度完成完整业务闭环。

语言：Python 框架：Django 5.2 LTS 数据库：SQLite 后台任务：数据库任务表 + 单Worker 部署：单机 / 单镜像

1. 技术方案结论

1个代码仓库页面、业务、AI、导出统一维护
1个数据库文件SQLite满足单机小规模使用
2个Python进程Web进程 + AI后台Worker
最终推荐：Django单体应用 + Django Templates + HTMX + SQLite + 本地文件存储 + 自定义数据库任务队列。
层级最终选型说明
语言Python 3.12+业务开发、AI调用、PDF解析、文档导出使用同一语言。
Web框架Django 5.2 LTS内置ORM、迁移、认证、权限、Session、表单、模板和管理后台。
页面Django Templates + HTMX + Bootstrap不建设独立Vue工程，降低前后端联调和部署成本。
数据库SQLite单机、小并发、低写入频率场景足够。
文件存储本地data目录简历、答卷、代码包和导出文件按任务目录存放。
后台任务ai_job表 + 单个Worker删除Redis、RabbitMQ、Celery。
模型调用统一LLMClient直接调用外部或私有模型API，结构化结果由Pydantic校验。
部署systemd或Docker Compose只启动Web和Worker两个进程。

2. 架构目标与边界

第一版目标

  • 用最少组件跑通完整招聘评测闭环。
  • 单人或小团队可以快速开发和维护。
  • 能够方便调试Prompt、模型输出和评分规则。
  • 支持普通题与现场开发题两条独立分支。
  • 后续可平滑升级数据库、存储和任务队列。

明确不建设

  • 微服务和服务注册发现。
  • 独立前端工程与Node生产服务。
  • Redis、RabbitMQ、MinIO、Keycloak。
  • Kubernetes、Prometheus、Grafana。
  • 在线IDE、自动代码运行、自动监考。
  • 复杂Agent框架与自主规划。

4. 技术选型说明

4.1 为什么选Django

能力Django内置支持本项目用途
ORM和迁移是任务、题目、提交、评分和报告数据管理。
用户认证是HR、技术负责人、面试官、管理员。
权限与用户组是基础RBAC与任务负责人权限。
表单和文件上传是创建任务、上传PDF、上传答卷和代码包。
模板渲染是直接实现现有六个核心页面。
管理后台是用户、模板、评分规则和异常数据维护。

4.2 为什么不使用FastAPI作为主框架

FastAPI适合纯API和独立AI服务，但本项目主要是内部管理页面、文件上传、状态流转、权限与后台配置。若采用FastAPI，仍需额外建设ORM迁移、认证、Session、管理后台和前端应用，整体复杂度更高。

4.3 为什么暂时使用SQLite

适用条件

  • 单机部署。
  • 一个Web实例和一个Worker。
  • 用户数量少，写入频率低。
  • AI任务不会长时间占用数据库事务。

使用要求

  • 开启WAL模式。
  • 事务保持短小。
  • Worker一次只领取一个任务。
  • 定期备份数据库文件和data目录。

6. AI后台任务机制

不引入RabbitMQ和Celery，使用数据库任务表与单个常驻Worker。

6.1 ai_job核心字段

字段说明
task_id关联招聘评测任务。
job_type分析、出题、评分、报告或导出。
statusqueued / running / success / failed / cancelled。
progress0—100，供前端轮询展示。
input_json任务输入快照或引用ID。
result_json结构化AI结果。
retry_count失败重试次数。
error_message错误摘要。

6.2 任务类型

analyze_position_resume generate_regular_questions generate_development_task score_regular_submission score_development_submission generate_report export_document

6.3 Worker流程

while True:
    job = claim_next_queued_job()

    if job is None:
        sleep(2)
        continue

    try:
        mark_running(job)
        result = execute_job(job)
        validate_result(result)
        save_result(job, result)
        mark_success(job)
    except Exception as exc:
        mark_failed_or_retry(job, exc)
并发要求：MVP只启动一个Worker。领取任务时使用短事务更新状态，避免两个进程重复执行同一任务。

7.3 文件规则

  • 数据库只保存相对路径，不保存BLOB。
  • 服务器保存名使用UUID，原始文件名单独记录。
  • 上传目录不直接暴露为公网静态目录。
  • 所有下载经过Django权限判断。
  • 简历、候选人答卷和用人部门答案包分目录存储。

格式技术用途

10. 用户权限与安全

10.1 角色

HR 技术负责人 面试官 管理员
角色主要权限
HR创建任务、上传简历、导出候选人版、记录发送与回收。
技术负责人确认待验证项、审核题目、查看答案和评分标准、确认评分。
面试官查看指定任务、录入评分和评价。
管理员用户、用户组、Prompt模板、评分规则和异常任务维护。

10.2 最低安全要求

  • 使用Django内置认证、CSRF、防XSS模板转义和Session安全配置。
  • 限制PDF、DOCX、XLSX、ZIP等允许上传类型和文件大小。
  • ZIP和候选人代码只存储，不在服务器自动执行。
  • 模型调用日志不保存完整个人联系方式和完整简历原文。
  • 下载、导出、发送、评分确认写入审计日志。
  • 生产环境配置HTTPS、Secure Cookie和定期备份。

13. 后续升级触发条件

出现的问题升级方案
多人同时写入，频繁出现database lockedSQLite迁移到PostgreSQL。
AI任务积压，需要多个Worker引入Redis + RQ，或RabbitMQ + Celery。
需要部署多个Web实例PostgreSQL + 共享对象存储。
本地文件容量或备份压力增大切换到MinIO或S3兼容存储。
需要企业统一登录接入LDAP、OIDC或Keycloak。
页面交互明显复杂化局部页面逐步引入Vue，不必全量重写。
需要复杂跨天流程与补偿再评估专业工作流引擎。
历史题库规模大、需要语义检索迁移PostgreSQL并增加pgvector。
演进原则：不为未来可能出现的问题提前增加中间件，只在真实瓶颈出现后替换对应组件。

15. 技术验收标准

  1. 单台服务器能够启动Web和Worker两个进程。
  2. 用户可登录并按角色访问任务数据。
  3. 可创建任务、上传文本型PDF简历并触发后台分析。
  4. 页面可查看任务进度和AI任务错误信息。
  5. 岗位、简历、待验证项和题目结果均通过结构化校验。
  6. 普通题可导出用人部门版和候选人版。
  7. 现场开发题可独立生成、发送、回收和评分。
  8. 候选人文件只能通过授权下载接口访问。
  9. 评分保留AI建议和人工最终结果。
  10. 系统能够生成最终评测报告并保留审计记录。
  11. 数据库与文件目录可一键备份和恢复。
  12. 在无Redis、RabbitMQ、PostgreSQL、MinIO的情况下完整跑通业务闭环。
